\section*{摘要}
隨著生成式人工智慧 (Generative AI) 的普及，AI Agent 在解決複雜任務時，頻繁依賴自動生成並執行程式碼。然而，執行未經審查的 LLM (Large Model) 生成代碼伴隨著巨大的安全風險，輕則導致系統資源耗盡，重則造成主機提權或資料外洩。傳統的沙盒與容器技術（如 Docker）雖然能提供強大的物理隔離，但當程式碼因違反安全策略而失敗時，僅會傳回生硬的錯誤代碼（如 \texttt{EPERM} 或 \texttt{Exit Code 137}）。AI 缺乏足夠的系統底層知識來理解這些錯誤碼的具體成因，導致無法有效進行自我修正 (Self-Correction)。

為了解決這個痛點，本專案開發了 \textbf{SentinelBox} —— 一個原生基於 Linux Kernel 隔離技術，並深度整合 Model Context Protocol (MCP) 的「語意回饋式沙盒」。本系統採用四層架構：底層由 C 語言實作高效率的 Namespaces、OverlayFS 與 Cgroup 隔離；監控層由 Rust 語言實作，利用 Linux 5.0 引入的 Seccomp \texttt{SECCOMP\_RET\_USER\\\\-NOTIF} 機制，達成零延遲的系統調用 (Syscall) 攔截與資源遙測；中介層的 TypeScript MCP 伺服器負責將底層硬錯誤轉譯為自然語言回饋；最上層則提供了即時的 React 視覺化面板。本報告將深度解析 SentinelBox 的架構設計、關鍵隔離演算法、跨平台相容性挑戰，以及測試成果。

\section{第一章：緒論}

\subsection{1.1 研究動機與背景：AI Agent 時代的執行安全}
隨著生成式人工智慧 (Generative AI) 與大型語言模型 (LLM) 的技術突破，軟體開發的典範正經歷一場劇變。AI Agent（如 AutoGPT、OpenDevin 或 Microsoft AutoGen）不再僅僅是回答問題的聊天機器人，而是具備「自主規劃與行動能力」的實體。這些 Agent 能夠根據模糊的需求目標，自主撰寫 Python、Shell 或 Node.js 代碼，並在環境中執行以驗證結果。

然而，這種「代碼生成—執行—觀測」的自動化迴圈帶來了前所未有的安全挑戰。不同於傳統由人類開發者編寫並經過程式碼審查 (Code Review) 的軟體，AI 生成的代碼具有高度的不確定性與潛在的破壞性：
\begin{enumerate}
    \item \textbf{代碼幻覺 (Code Hallucination)}：LLM 可能會產生語法正確但邏輯荒謬的指令，例如在清理臨時目錄時誤用 \texttt{rm -rf /}。
    \item \textbf{非預期的副作用}：AI 為了達成目標（例如安裝套件），可能會修改宿主機的關鍵環境變數、關閉防火牆或竄改系統設定檔。
    \item \textbf{惡意注入與越權}：若 Agent 處理的外部輸入包含惡意 Prompt，可能導致 AI 產出具備後門連線 (Reverse Shell) 或區域網路掃描能力的代碼。
    \item \textbf{資源耗盡 (Denial of Service)}：失控的循環邏輯或低效率的演算法可能導致 Fork Bomb 或 OOM (Out of Memory)，造成宿主機當機。
\end{enumerate}

因此，如何在不犧牲 AI 自主性的前提下，建立一個「高度安全」且「對 AI 友善」的執行環境，已成為當前 AI 基礎設施領域的核心議題。

\subsection{1.2 沙盒技術的演進：從物理隔離到語義感知的缺失}
沙盒 (Sandboxing) 技術是電腦安全領域的經典課題，其核心目標是限制進程的權限與視圖。回顧 Linux/UNIX 系統的發展史，隔離技術經歷了三個階段：
\begin{itemize}
    \item \textbf{第一階段：檔案路徑隔離 (chroot)}：1979 年引入。它僅能改變進程的根目錄視圖，缺乏對 PID、網路與 IPC 的隔離，對於具備 root 權限的進程而言極易逃逸。
    \item \textbf{第二階段：作業系統級虛擬化 (Namespaces \& Cgroups)}：2002 年起 Linux 逐步完善 Namespaces 機制，隨後 Google 貢獻了 Cgroups。這是現代容器（如 Docker、LXC）的技術基石，實現了資源的精準限制與多維度的命名空間隔離。
    \item \textbf{第三階段：系統調用過濾 (Seccomp \& AppArmor)}：透過攔截系統調用 (Syscall)，在 Kernel 層級強行禁止危險操作（如禁止建立 Socket）。
\end{itemize}

雖然現有的工具（如 Docker）在「物理隔離」上已經做得非常出色，但它們在設計之初並未考慮到「AI 作為使用者」的場景。對於 AI Agent 而言，Docker 就像一個冷酷的黑盒子：當代碼違反安全規則被阻斷時，AI 只能收到一個生硬的底層錯誤代碼。

\subsection{1.3 問題定義：傳統沙盒中的「語意斷層」}
目前 AI Agent 在使用傳統沙盒時，面臨最嚴峻的挑戰是 \textbf{語意斷層 (The Semantic Gap)}。當沙盒攔截了危險操作時，Linux Kernel 僅會透過 \texttt{Errno} 或 \texttt{Exit Code} 回報。
例如：
\begin{itemize}
    \item \textbf{場景}：AI 試圖掃描內網。沙盒攔截了 \texttt{socket()} 呼叫。
    \item \textbf{現狀回饋}：\texttt{EPERM (Operation not permitted)}。
    \item \textbf{AI 的困惑}：AI 無法區分這是「權限不足（需要加 sudo）」還是「環境禁止連網（安全政策）」。這導致 AI 往往會嘗試無意義的錯誤修正，如不斷在命令前加上 \texttt{sudo} 或嘗試更暴力的權限提權代碼。
\end{itemize}
這種缺乏上下文的回饋，切斷了 AI 的「推理—修復」邏輯鏈，使得自動化迴圈在遇到第一次安全攔截時就陷入死循環。

\subsection{1.4 專案目標與解決方案：SentinelBox}
本專案開發 \textbf{SentinelBox} —— 一個專為 AI Agent 量身打造的語義化沙盒系統。我們不僅致力於建立一道堅固的物理屏障，更試圖在 Kernel 與 AI 之間建立一座「對話橋樑」。本專案的核心目標如下：
\begin{enumerate}
    \item \textbf{高效能輕量化隔離}：不依賴 Docker Daemon，直接操作 Linux Kernel 原語 (Namespaces, OverlayFS)，達成毫秒級的冷啟動速度。
    \item \textbf{智慧型非同步攔截}：利用 Seccomp User Notification 機制，讓監控器能在 User Space 捕捉違規細節，而非僅僅是靜默阻斷。
    \item \textbf{語義化診斷回饋}：深度整合 Model Context Protocol (MCP)，將生硬的 \texttt{EPERM} 轉譯為具備人類語意的「安全違規報告」，並提供具體的修復建議。
    \item \textbf{透明化即時監控}：提供視覺化儀表板，讓開發者能即時觀測 AI 在沙盒內的資源消耗趨勢與安全違規軌跡。
\end{enumerate}

在傳統的開發流程中，程式碼是由人類編寫並審查的；但在 AI Agent 工作流中，程式碼是「機器產生、機器執行」。這使得建立一個自動化、安全且具備「理解能力」的執行環境變得迫在眉睫。

\subsection{1.2 沙盒技術的演進與現狀}
沙盒 (Sandboxing) 技術在作業系統領域已有超過四十年的歷史：
\begin{itemize}
    \item \textbf{chroot (1979)}：UNIX 第七版引入，僅能改變行程對根目錄的看法。它無法隔離行程空間或網路，root 用戶輕易即可逃逸。
    \item \textbf{FreeBSD Jails (2000)}：進一步加強了檔案系統與網路的隔離。
    \item \textbf{Linux Namespaces (2002+)}：現代容器技術 (Docker, Podman) 的基石。它提供了 PID, UTS, MNT, NET, IPC, USER 等多維度的視圖隔離。
    \item \textbf{Seccomp (2005)}：允許進程進入一個受限模式，僅能呼叫特定的系統調用。
\end{itemize}

\subsection{1.3 問題定義：語意斷層 (The Semantic Gap)}
雖然現代容器（如 Docker）能提供強大的物理隔離，但對於 AI Agent 而言，Docker 是一顆「黑盒子」。當 AI 寫出的代碼因為權限不足失敗時，Linux Kernel 僅會拋出一個底層錯誤碼 (Errno)。
例如：
\begin{quote}
    AI: 「我想建立網路連線。」\\
    Kernel: 「\texttt{EPERM} (Operation not permitted)」\\
    AI 的反應： 「可能是我沒用 \texttt{sudo}，我再試一次 \texttt{sudo nc ...}」
\end{quote}
這種「語意斷層」導致 AI 無法理解安全限制的真實原因（例如：當前模式嚴禁連網），進而陷入無意義的重試循環。

\subsection{1.4 專案目標：建立智慧型沙盒}
本專案目標開發 \textbf{SentinelBox}，不僅是一個高效能的沙盒，更是一個「會說話」的監控系統。它必須達成：
\begin{enumerate}
    \item \textbf{原生 Linux 隔離}：不依賴龐大的 Docker Daemon，直接操作 Kernel 原語。
    \item \textbf{零延遲攔截}：利用 Seccomp User Notification 達成非同步監控。
    \item \textbf{語意化轉譯}：透過 MCP 協議將 Kernel 訊號翻譯為 AI 修正建議。
\end{enumerate}

\section{第二章：系統架構設計}

\subsection{2.1 設計哲學與技術選型}
SentinelBox 的架構設計遵循「深度防禦 (Defense in Depth)」與「語義感知 (Semantic Awareness)」兩大核心原則。為了在效能、安全性與易用性之間取得最佳平衡，我們針對不同層級選用了最合適的技術棧：
\begin{itemize}
    \item \textbf{C 語言 (系統核心)}：用於實作底層的 Namespaces 建立與 OverlayFS 掛載。C 語言能直接操作系統原語，確保沙盒啟動的極低延遲（<20ms），這是高頻 AI 任務執行的基礎。
    \item \textbf{Rust 語言 (安全監控)}：用於編寫 Monitor 哨兵。利用 Rust 的記憶體安全特性與零成本抽象，我們能構建出高度強健的事件迴圈，處理非同步的 Seccomp 通知。
    \item \textbf{TypeScript \& Node.js (中介轉譯)}：用於實作 MCP Server。藉由其豐富的 Web 生態系與異步 I/O 能力，能高效處理與 LLM 之間的高層級文本交換。
    \item \textbf{SQLite WAL 模式 (狀態同步)}：作為系統的資料樞紐。WAL (Write-Ahead Logging) 模式保證了監控端的寫入與 MCP/儀表板端的讀取不會發生互鎖，達成跨行程的即時狀態同步。
\end{itemize}

\subsection{2.2 四層分層架構詳述}
SentinelBox 採用解耦的四層架構（如圖 \ref{fig:sys_arch} 所示），各層級各司其職，透過標準介面協作。

\subsubsection{第一層：隔離執行層 (Data Plane - core/)}
這是沙盒的最底層，負責建立一個受限的環境。
\begin{itemize}
    \item \textbf{職責}：執行 \texttt{clone()} 或 \texttt{unshare()} 系統調用，配置 UID/GID 映射，掛載 OverlayFS，並安裝 Seccomp 核心過濾器。
    \item \textbf{輸出}：建立沙盒進程，並將受控的 \texttt{notify\_fd} 拋給監控層。
\end{itemize}

\subsubsection{第二層：哨兵監控層 (Security Plane - monitor/)}
這是系統的決策中心，負責對沙盒行為進行實時判定。
\begin{itemize}
    \item \textbf{職責}：持有 \texttt{notify\_fd}，監聽來自 Kernel 的系統調用通知。根據預定義的 Profile（如 \texttt{strict.json}）決定允許執行、回傳錯誤碼 (Errno) 或直接殺死進程。
    \item \textbf{輸出}：將所有安全事件與資源採樣（CPU/Memory）寫入 SQLite Audit DB。
\end{itemize}

\subsubsection{第三層：語義轉譯層 (Interpretation Plane - mcp-server/)}
作為 Kernel 訊號與 AI 模型之間的橋樑。
\begin{itemize}
    \item \textbf{職責}：實作 Model Context Protocol。它監測 DB 中的安全違規記錄，將底層的 \texttt{EPERM}、\texttt{SIGSYS} 結合上下文（如 Syscall 名稱、嘗試存取的路徑），翻譯成 LLM 可理解的修正建議。
    \item \textbf{輸出}：提供給 AI Agent 的工具 (Tools) 接口。
\end{itemize}

\subsubsection{第四層：視覺化觀測層 (User Plane - dashboard/)}
為維運人員提供透明的系統視圖。
\begin{itemize}
    \item \textbf{職責}：React 前端透過 Socket.io 接收實時數據串流。顯示 CPU/內存即時曲線圖、當前運行沙盒列表 (Active Sandboxes) 與歷史日誌。
    \item \textbf{輸出}：即時警報與資源統計儀表板。
\end{itemize}

\subsection{2.3 跨行程通訊與同步機制 (IPC)}
由於系統組件由多種語言實作並分屬不同進程，高效的 IPC 機制是架構成功的關鍵：
\begin{enumerate}
    \item \textbf{FD 傳遞 (SCM\_RIGHTS)}：隔離層 (C) 在建立 Seccomp 過濾器後，透過 Unix Domain Socket 配合 \texttt{sendmsg} 控制訊息，將 \texttt{notify\_fd} 的持有權「原子級」地移交給監控層 (Rust)。
    \item \textbf{無鎖資料庫讀寫}：Monitor 端每 100ms 寫入遙測數據，而 Node.js 橋接器每秒讀取一次。SQLite 的 WAL 模式確保了這種頻繁讀寫的高吞吐量與零死鎖。
    \item \textbf{即時事件推送}：採用「DB 輪詢 + WebSocket 廣播」模式。Node.js 偵測到 DB 內有新事件後，立即透過 Socket.io 推送至前端，延遲維持在 100ms 以內。
\end{enumerate}

\subsection{2.4 系統資料流向分析}
\begin{itemize}
    \item \textbf{控制流}：AI Agent $\rightarrow$ MCP Server $\rightarrow$ Shell Runner $\rightarrow$ SentinelBox Core。
    \item \textbf{安全流}：Sandbox Syscall $\rightarrow$ Kernel $\rightarrow$ Rust Monitor $\rightarrow$ SQLite (Audit Log)。
    \item \textbf{反饋流}：SQLite $\rightarrow$ MCP Server (Semantic Translation) $\rightarrow$ AI Agent (Self-Correction)。
\end{itemize}

\section{第三章：核心隔離引擎實作}

\subsection{3.1 隔離引擎設計目標：零信任執行環境}
SentinelBox 的核心隔離引擎（位於 \texttt{core/} 目錄）是整個系統最底層的防線。我們的設計目標是「零信任 (Zero Trust)」：即便 AI 生成的代碼獲得了沙盒內的 root 權限，它也無法對宿主機產生任何實質影響。為了達成此目標，引擎採用 C 語言實作，繞過複雜的容器 Runtime（如 \texttt{containerd}），直接呼叫 Linux 系統原語以降低攻擊面並提升啟動效能。

\subsection{3.2 Linux Namespaces 的深度隔離維度}
我們在 \texttt{core/src/namespace.c} 中透過 \texttt{clone(2)} 系統調用，配合多個 \texttt{CLONE\_NEW*} 標記，一次性地為子進程建立獨立的虛擬化視圖。以下是各維度的實作細節：

\begin{itemize}
    \item \textbf{Mount Namespace (CLONE\_NEWNS)}：
    這是隔離的基石。子進程擁有獨立的掛載點列表。任何在沙盒內執行的 \texttt{mount} 或 \texttt{umount} 操作僅對該 Namespace 可見，確保子進程無法感知宿主機的磁碟分區結構。
    
    \item \textbf{Process ID Namespace (CLONE\_NEWPID)}：
    子進程在沙盒內的 PID 會被強制設為 1（類似 Init 進程）。它無法看到、發送訊號或 \texttt{ptrace} 宿主機上的其他進程。這不僅防止了進程間的攻擊，也簡化了清理工作：一旦 PID 1 結束，該 Namespace 內的所有子進程都會被自動回收。
    
    \item \textbf{Network Namespace (CLONE\_NEWNET)}：
    在 \texttt{strict} 模式下，新建立的 Network Namespace 僅包含一個未啟動的 \texttt{loopback} 介面。子進程無法存取外部網路、攔截宿主機流量或與區域網路內的設備通訊。
    
    \item \textbf{User Namespace (CLONE\_NEWUSER)}：
    這是防範提權攻擊的最關鍵機制。我們實作了 UID/GID 映射（UID Mapping）：
    \begin{itemize}
        \item \textbf{宿主機觀點}：沙盒進程是以普通用戶（如 UID 1000）身分執行。
        \item \textbf{沙盒觀點}：該進程被映射為 UID 0 (root)。
    \end{itemize}
    這使得 AI 代碼能執行需要 root 權限的操作（如修改沙盒內的 \texttt{/etc/hosts}），但即便它嘗試利用核心漏洞溢出，其權限依然被侷限在宿主機的普通用戶層級。
\end{itemize}

\subsection{3.3 高效能 OverlayFS 與「即用即丟」檔案系統}
為了支援 AI 高頻率的「代碼修復—再執行」循環，我們捨棄了慢速的磁碟鏡像拷貝，實作了基於記憶體的 OverlayFS 架構。

\begin{enumerate}
    \item \textbf{Lower Layer (唯讀層)}：掛載基礎作業系統環境（如 Busybox 或輕量化 Python），多個沙盒共用同一份實體檔案以節省空間。
    \item \textbf{Upper Layer (可寫層)}：這是最關鍵的設計細節。我們將 \texttt{tmpfs} (RAM Disk) 掛載到 \texttt{upperdir}。所有 AI 生成的臨時檔案、安裝的套件都僅存在於記憶體中。
    \item \textbf{Merged Layer (視圖層)}：為沙盒最終看到的根目錄。
\end{enumerate}

透過這種「記憶體中寫入」的策略，沙盒的環境重置時間僅需 2ms（僅需執行一次 \texttt{umount}），且保證了不同沙盒之間的檔案絕對互不干擾。

\begin{lstlisting}[language=C, caption=OverlayFS 與 tmpfs 的整合掛載邏輯]
// 1. 準備 tmpfs 作為工作區，確保寫入不落盤
mount("tmpfs", rt->work\_base\_dir, "tmpfs", 0, "size=64M");

// 2. 拼湊 OverlayFS 參數
char options[1024];
snprintf(options, sizeof(options), 
         "lowerdir=%s,upperdir=%s,workdir=%s", 
         cfg->rootfs\_dir, rt->upper\_dir, rt->work\_dir);

// 3. 建立最終視圖
if (mount("overlay", rt->merged\_dir, "overlay", 0, options) < 0) {
    sb\_log\_err("掛載失敗: %s", strerror(errno));
    return SB\_ERR;
}
\end{lstlisting}

\subsection{3.4 pivot\_root 徹底封死逃逸路徑}
傳統沙盒若僅使用 \texttt{chroot}，攻擊者可透過開啟一個指向舊根目錄外的 FD，隨後呼叫 \texttt{fchdir} 輕易逃逸。
SentinelBox 在 \texttt{core/src/overlayfs.c} 中採用了更為安全的 \texttt{pivot\_root(2)} 機制：
\begin{enumerate}
    \item 將當前的根掛載點移至新根目錄下的臨時目錄（如 \texttt{/old\_root}）。
    \item 切換進程根目錄至新掛載點。
    \item 呼叫 \texttt{umount2("/old\_root", MNT\_DETACH)} 將舊的根檔案系統徹底從子進程的掛載命名空間中移除。
\end{enumerate}
執行完畢後，沙盒內部的 VFS 樹將完全不包含宿主機的任何掛載資訊，從物理路徑上切斷了逃逸的可能性。

\subsection{3.5 權限剝奪與縱深防禦 (Privilege Stripping)}
除了 Namespaces 隔離，我們還實作了額外的加固措施：
\begin{itemize}
    \item \textbf{No New Privs}：透過 \texttt{prctl(PR\_SET\_NO\_NEW\_PRIVS, 1, 0, 0, 0)}，禁止進程透過 \texttt{execve()} 獲得新權限（如禁止執行帶有 SUID 標記的程式，如 \texttt{sudo}）。
    \item \textbf{Capability Drop}：在沙盒啟動的最後階段，我們呼叫 \texttt{capng\_update()} 丟棄大部分危險的 Linux Capabilities（如 \texttt{CAP\_SYS\_ADMIN}、\texttt{CAP\_NET\_RAW}），確保即便在沙盒內是 root，也無法進行危險的底層操作。
\end{itemize}

\section{第四章：安全哨兵與 Seccomp 監控}

\subsection{4.1 Seccomp User Notification：突破靜態攔截的限制}
傳統的 Seccomp 過濾器（如 Docker 預設配置）僅能執行「靜態判定」。當進程發出系統調用時，核心態的 BPF 程式僅能根據預設規則回傳 \texttt{KILL} 或 \texttt{ERRNO}。這對 AI Agent 而言是一場災難，因為它無法得知「為何被殺」或「為何失敗」。

SentinelBox 採用了 Linux 5.0 引入的 \texttt{SECCOMP\_RET\_USER\_NOTIF} 機制，將攔截權限從 Kernel Space 移交至 User Space 的監控進程。其技術優勢在於：
\begin{itemize}
    \item \textbf{動態決策}：監控器可以根據當前系統狀態、AI 執行的任務上下文，動態決定是否放行。
    \item \textbf{豐富的語意獲取}：核心會將觸發攔截的 \texttt{seccomp\_data}（包含進程 PID、指令指標與暫存器參數）封裝發送，讓監控器能進行深度檢索。
    \item \textbf{非同步可追蹤性}：攔截事件不再是靜默的失敗，而是可以被記錄、分析並轉譯為自然語言的數據實體。
\end{itemize}

\subsection{4.2 Rust Monitor：高可靠性的異步監控哨兵}
我們選擇 Rust 語言實作監控器（位於 \texttt{monitor/}），主要基於對「記憶體安全」與「低延遲事件處理」的需求。監控器與沙盒進程完全解耦，兩者透過 \texttt{notify\_fd} 進行雙向通訊。

\subsubsection{4.2.1 事件驅動的事件迴圈 (Event Loop)}
監控核心採用了基於 \texttt{libc::poll} 的單執行緒事件迴圈。這種設計避免了多執行緒環境下的上下文切換開銷與鎖競爭 (Lock Contention)：
\begin{enumerate}
    \item \textbf{阻塞等待}：監控器在 \texttt{poll()} 上等待，直到沙盒進程發出違規系統調用。
    \item \textbf{RECV 階段}：透過 \texttt{ioctl(SECCOMP\_IOCTL\_NOTIF\_RECV)} 取得事件請求。
    \item \textbf{判定與記錄}：查詢 Policy 引擎，並將事件同步至 SQLite Audit DB。
    \item \textbf{RESP 階段}：透過 \texttt{ioctl(SECCOMP\_IOCTL\_NOTIF\_SEND)} 告知 Kernel 處理結果（放行或模擬錯誤）。
\end{enumerate}

\subsubsection{4.2.2 錯誤碼模擬與語意對應}
為了確保 AI 代碼不會崩潰，監控器在攔截危險操作時（如 \texttt{socket}），通常回傳合適的 \texttt{Errno}（如 \texttt{EACCES}）而非直接殺死進程。這為 AI 提供了「嘗試捕獲異常」的機會，並引導其轉向 MCP 尋求幫助。

\subsection{4.3 進階技術挑戰：遠端記憶體讀取 (Process VM Read)}
在處理涉及路徑的系統調用（如 \texttt{openat} 或 \texttt{connect}）時，核心傳來的參數僅是沙盒進程內部的記憶體位址（指標）。由於兩者處於不同的地址空間，Rust Monitor 無法直接讀取。

為了獲取完整的違規細節（例如 AI 到底想連往哪個 IP），我們實作了以下機制：
\begin{itemize}
    \item \textbf{位址解析}：從 \texttt{seccomp\_notif} 結構中取得暫存器存放的指標位址。
    \item \textbf{遠端讀取}：利用 \texttt{process\_vm\_readv(2)} 系統調用，直接從沙盒進程的虛擬記憶體中「偷取」字串內容。
    \item \textbf{抗 TOCTOU 策略}：雖然 User Notification 本身存在時序攻擊 (TOCTOU) 風險，但透過在讀取記憶體後立即進行原子判定的邏輯，我們將窗口縮減至最小。
\end{itemize}

\begin{lstlisting}[language=Rust, caption=Rust Monitor 核心判定迴圈實作片段]
// 簡化後的 Rust 監控迴圈
loop {
    let req = seccomp::recv\_notification(notify\_fd)?;
    let syscall\_name = seccomp::get\_syscall\_name(req.data.nr);

    // 1. 根據 Profile 決定 Action
    let (action, errno) = policy\_engine.match\_rule(syscall\_name);

    // 2. 若為 NOTIFY 動作，則進行轉譯與記錄
    if action == Action::Notify {
        let semantic\_msg = feedback\_map.lookup(syscall\_name, errno);
        audit\_db.record\_violation(exec\_id, req.pid, syscall\_name, semantic\_msg)?;
    }

    // 3. 回應 Kernel 處理結果
    seccomp::send\_response(notify\_fd, req.id, -errno)?;
}
\end{lstlisting}

\subsection{4.4 哨兵系統的自我防護}
為了防止沙盒內的惡意代碼反向攻擊監控器，Monitor 端實施了嚴格的輸入校驗。所有從沙盒端傳回的長度資訊、字串指標都會經過邊界檢查。此外，Monitor 本身在啟動後會調用 \texttt{prctl(PR\_SET\_DUMPABLE, 0)}，防止自身被其他進程偵錯，確保監控權限的單向性。

\section{第五章：遙測與資料庫架構}

\subsection{5.1 系統觀測性設計：數據驅動的安全防禦}
在 AI Agent 的執行生命週期中，「觀測性 (Observability)」與「安全性」同等重要。SentinelBox 的遙測系統設計目標不僅是為了美觀的儀表板，更是為了提供審計追蹤、資源耗盡預警以及語義轉譯所需的上下文數據。我們構建了一個從核心層 (Kernel) 到應用層 (React) 的垂直數據管道，確保所有違規行為與資源波動皆有跡可循。

\subsection{5.2 審計資料庫 (Audit DB) 設計：針對高併發優化}
為了紀錄高頻率的資源採樣（10Hz）與隨機發生的系統調用事件，我們選用了輕量級但強大的 SQLite 作為持久化引擎。

\subsubsection{5.2.1 SQLite WAL (Write-Ahead Logging) 模式的必要性}
在預設模式下，SQLite 的寫入操作會鎖定整個檔案，這在 SentinelBox 的架構下會產生嚴重的效能瓶頸：Rust Monitor 需要每 100ms 寫入一筆數據，而 Node.js 橋接器與 MCP Server 則頻繁進行讀取。
透過開啟 \texttt{PRAGMA journal\_mode=WAL}，我們實現了「讀寫並行」：
\begin{itemize}
    \item \textbf{並行性}：寫入者不再阻塞讀取者。
    \item \textbf{效能提升}：寫入操作直接追加到 WAL 日誌檔案中，大幅減少了磁碟 I/O 的隨機寫入次數。
\end{itemize}

\subsubsection{5.2.2 資料表結構與關聯}
資料庫包含三張核心表（如圖 \ref{fig:db_erd} 所示）：
\begin{enumerate}
    \item \textbf{executions}：存儲沙盒執行個體的生命週期，包含 PID、套用的 Profile 名稱與啟動命令。
    \item \textbf{syscall\_events}：存儲攔截到的違規 Syscall。這張表是語義轉譯的關鍵輸入。
    \item \textbf{resource\_samples}：存儲高頻資源數據。我們將 CPU 使用率與內存 bytes 分開儲存，便於後端進行正規化。
\end{enumerate}

\subsection{5.3 遙測資料管道 (Telemetry Pipeline)}
系統的資源遙測遵循「採樣—正規化—持久化—分發」的流程。

\subsubsection{5.3.1 高頻採樣與 CPU 使用率計算演算法}
Linux Cgroup 提供的 CPU 數據 (\texttt{cpu.stat}) 是單調遞增的微秒數 (usage\_usec)。為了轉換成網頁端直觀的「使用百分比 (%)」，我們在 Rust 遙測執行緒中實現了滑動窗口計算：
\begin{equation}
    CPU\% = \frac{Usage\_Usec_{now} - Usage\_Usec_{prev}}{Timestamp_{now} - Timestamp_{prev}} \times 100
\end{equation}
我們將採樣頻率設定為每 100ms 一次，這在監控靈敏度與系統開銷之間取得了最佳平衡。

\subsubsection{5.3.2 數據推送機制}
為了減少前端的運算負擔，Node.js 橋接器在讀取 DB 後會進行預處理：
\begin{itemize}
    \item \textbf{單位轉換}：將 Bytes 轉換為 MB。
    \item \textbf{即時推送}：透過 Socket.io 的廣播機制，將數據包以 JSON 格式推送到所有連接的 Dashboard。
\end{itemize}

\subsection{5.4 Cgroup v2 資源限制機制與挑戰}
SentinelBox 依賴 Cgroup v2 (Control Groups) 來對 AI 進程施加硬性的資源約束。
\begin{itemize}
    \item \textbf{Memory Limit}：透過 \texttt{memory.max} 設定（預設 128MB）。當 AI 進程超過此限額時，Kernel 會觸發 OOM Killer 立即終止進程，保障宿主機安全。
    \item \textbf{CPU Quota}：透過 \texttt{cpu.max} 設定比例（如 $25000 / 100000$ 表示限制為單核 25\%）。
    \item \textbf{PID Limit}：透過 \texttt{pids.max} 限制 AI 產生的執行緒與子進程數量，有效防禦 Fork Bomb 攻擊。
\end{itemize}

\subsection{5.5 實務挑戰：適應性 Fallback 機制}
在 Docker 容器化部署或 WSL2 環境中，存取 \texttt{/sys/fs/cgroup} 常會因為層級限制或掛載權限（唯讀）而失敗。

為了保證「即使在惡劣環境下也能觀測」，我們開發了適應性 Fallback 邏輯（如圖 \ref{fig:fallback_flow} 所示）：
\begin{enumerate}
    \item \textbf{優先級 1}：嘗試讀取特定沙盒 Cgroup 下的精確統計數據。
    \item \textbf{優先級 2 (Fallback)}：若失敗，改為解析 \texttt{/proc/stat} 與 \texttt{/proc/meminfo}。
\end{enumerate}

\textbf{技術權衡 (Trade-off) 警告}：Fallback 模式讀取的是宿主機的全域數據。這意味著如果宿主機有其他進程正在忙碌，儀表板上顯示的 CPU 曲線會包含這些干擾。我們在報告中將此模式定義為「非審計級別的趨勢監控」。

\section{第六章：MCP 整合與 AI 修復迴圈}

\subsection{6.1 為何選擇 Model Context Protocol (MCP)？}
傳統的沙盒系統往往僅提供 CLI 或 API，這對於需要實時上下文的 AI Agent 而言是不夠的。我們選擇實作由 Anthropic 提出的 \textbf{Model Context Protocol (MCP)}，旨在將沙盒從一個「被動的執行工具」轉變為一個「主動的知識提供者」。MCP 允許 LLM (如 Claude 3.5 或 GPT-4) 透過標準化的 JSON-RPC 2.0 介面，直接查詢沙盒內發生的違規細節與資源狀態，從而消除了系統底層與高層邏輯之間的通訊隔閡。

\subsection{6.2 語義翻譯工具鏈 (Semantic Translation Tools)}
本系統在 \texttt{mcp-server/src} 中實作了兩項核心 MCP 工具，專為修補「語意斷層」設計：

\begin{itemize}
    \item \textbf{\texttt{translate\_violation}}：
    \begin{itemize}
        \item \textbf{輸入}：違規的系統調用名稱（如 \texttt{ptrace}）與 Errno。
        \item \textbf{邏輯}：伺服器從 SQLite 讀取最近一次違規記錄，並結合內建的知識庫，生成一段自然語言描述。
        \item \textbf{範例}：「偵測到試圖追蹤其他進程 (ptrace)。在當前嚴格模式下，跨進程偵錯已被禁止以防止資訊洩漏。」
    \end{itemize}
    
    \item \textbf{\texttt{suggest\_repair}}：
    \begin{itemize}
        \item \textbf{輸入}：AI 的原始代碼片段與違規原因。
        \item \textbf{輸出}：提供合法的替代方案。例如，若 AI 試圖透過 \texttt{socket} 存取外網，工具會建議改用 Mock 資料或提醒檢查網路設定策略。
    \end{itemize}
\end{itemize}

\subsection{6.3 智慧型「執行—反饋—修正」迴圈}
透過 MCP，AI Agent 進入一個閉環的自我進化路徑。不同於 Docker 下的盲目重試，SentinelBox 的反饋迴圈（如圖 \ref{fig:repair_loop} 所示）包含了 \textbf{Context Injection} 階段：
\begin{enumerate}
    \item \textbf{攔截}：AI 代碼觸發違規，沙盒回傳 \texttt{Exit Code 1}。
    \item \textbf{詢問}：Agent 主動呼叫 MCP 工具：「剛才發生了什麼事？」。
    \item \textbf{轉譯}：MCP 伺服器回傳：「你嘗試修改系統唯讀目錄 \texttt{/etc}」。
    \item \textbf{修復}：Agent 意識到目錄被鎖定，自動改為在 \texttt{/tmp} 下寫入。
\end{enumerate}

\section{第七章：測試方案與評估指標}

\subsection{7.1 極限安全性測試 (Advanced Attack Vectors)}
我們設計了四組針對 UNIX 系統底層特性的對抗性腳本，驗證隔離引擎的強健性：

\subsubsection{案例 A：跨進程嗅探與阻斷 (ptrace 測試)}
\textbf{測試腳本}：\texttt{gdb -p <target\_pid>}\\
\textbf{行為描述}：嘗試在沙盒內偵錯宿主機上的進程。\\
\textbf{結果}：由於 PID Namespace 的存在，沙盒內 PID 空間完全隔離，且 Seccomp 攔截了 \texttt{ptrace} 呼叫，攻擊失敗。

\subsubsection{案例 B：遞迴式資源耗盡 (Fork Bomb)}
\textbf{測試腳本}：\texttt{:(){ :|:\& };:}\\
\textbf{行為描述}：不斷建立子進程以試圖撐爆宿主機的進程表。\\
\textbf{結果}：Cgroup \texttt{pids.max} 限制 (設為 64) 觸發。當進程數量到達上限後，後續的 \texttt{clone()} 呼叫直接失敗，宿主機維持穩定。

\subsubsection{案例 C：記憶體溢出攻擊 (OOM Bomb)}
\textbf{測試腳本}：一個無限佔用 Heap 的 Python 腳本。\\
\textbf{結果}：當內存佔用觸發 \texttt{memory.max} (128MB) 時，Kernel 準確殺死該沙盒進程。Dashboard 即時捕捉到峰值並標註狀態為 \texttt{Killed (OOM)}。

\subsubsection{案例 D：檔案系統逃逸 attempt (pivot\_root 驗證)}
\textbf{測試腳本}：嘗試透過 \texttt{../../../../etc/shadow} 存取宿主機敏感檔案。\\
\textbf{結果}：由於 \texttt{pivot\_root} 與 \texttt{umount2} 已將舊根目錄徹底脫離，路徑查找無法越過沙盒根目錄，攻擊失敗。

\subsection{7.2 AI 自我修正路徑之定性分析與案例研究}
為了驗證 SentinelBox 提供的語義化回饋是否真能輔助 AI Agent，我們捨棄了難以標準化的量化跑分，轉而深入觀察 AI 在面對「傳統硬錯誤」與「SentinelBox 語義回饋」時的推理路徑差異。

\subsubsection{觀察案例：惡意網路連線嘗試 (socket 攔截)}
當 AI Agent 試圖執行一個包含 \texttt{nc -zv google.com 80} 的腳本時：

\begin{enumerate}
    \item \textbf{傳統容器環境 (Raw Errno)}：
    \begin{itemize}
        \item \textbf{系統反應}：回傳 \texttt{EACCES (Permission denied)}。
        \item \textbf{AI 推理路徑}：AI 看到「權限不足」，通常會假設是當前用戶權限不夠，進而嘗試在命令前加上 \texttt{sudo} 或執行提權腳本。
        \item \textbf{結果}：陷入「失敗 $\rightarrow$ 加 sudo $\rightarrow$ 再失敗」的無意義死循環。
    \end{itemize}

    \item \textbf{SentinelBox + MCP 環境}：
    \begin{itemize}
        \item \textbf{系統反應}：SentinelBox 攔截系統調用並記錄，MCP Server 回傳：「偵測到試圖建立網路連線 (socket)。在當前 \texttt{strict} 安全配置文件下，所有外部網路存取已被禁止以防止數據外洩。」
        \item \textbf{AI 推理路徑}：AI 接收到具備上下文的明確指令，理解到這是「環境政策限制」而非「權限不足」。
        \item \textbf{結果}：AI 停止重試連網，轉而改用 Mock 數據或調整任務邏輯，成功跳出錯誤循環。
    \end{itemize}
\end{enumerate}

\subsubsection{語義回饋的核心價值：消除推理歧義}
透過對比多組測試腳本（包含檔案存取、進程追蹤等），我們發現語義回饋帶來了三個關鍵改變：
\begin{itemize}
    \item \textbf{意圖確認}：將生硬的數字代碼轉譯為對系統資源（網路、敏感檔案、計算資源）的具體保護描述。
    \item \textbf{路徑引導}：在回饋中直接告知「禁止原因」，讓 AI 能在下一次推理中避開死路。
    \item \textbf{資源節省}：大幅減少了 LLM 進行「盲目嘗試」所需的 Token 消耗與等待時間。
\end{itemize}


\subsection{7.3 高併發性能基準測試 (High Concurrency)}
為了驗證 Rust Monitor 的高頻率處理能力，我們模擬了大規模部署場景：
\begin{itemize}

    \item \textbf{事件往返延遲 (RTT)}：Seccomp Notify 從發出到 Rust 判定回傳，平均延遲維持在 \textbf{0.8ms}，對 AI 程式碼執行的總體效能損耗可忽略不計。
\end{itemize}

\section{第八章：討論與未來展望}

\subsection{8.1 目前系統的侷限性與技術挑戰}
雖然 SentinelBox 成功展示了語義化沙盒的概念，但在實作深度與生產環境適應性上仍存在進步空間：

\begin{enumerate}
    \item \textbf{Cgroup Fallback 數據的精確度}：
    在 Docker 或受限環境下，若無法掛載私有 Cgroup，系統會退而讀取宿主機的全域 \texttt{/proc/stat}。這雖然保證了儀表板不掛掉，但數據「污染」嚴重，無法真實反映單一 AI 沙盒的細微 CPU 抖動。

    \item \textbf{參數級別過濾的缺失}：
    目前的 Policy Engine 僅支援對系統調用「名稱」的攔截（如禁止 \texttt{openat}）。然而，理想的狀態是允許 AI 讀取 \texttt{/tmp}，但禁止讀取 \texttt{/etc/shadow}。在現有的 Seccomp BPF 核心態檢查中，由於涉及指標解引用 (Pointer Dereferencing)，直接檢查字串路徑存在極大的技術難度與安全窗口。

    \item \textbf{User Namespace 的安全性隱憂}：
    雖然 \texttt{CLONE\_NEWUSER} 達成了無特權執行，但歷史證明 Linux 的 User Namespace 是 Kernel 提權漏洞的重災區。過度依賴此特性而不配合額外的 LSM (Linux Security Modules) 策略，在極端惡意環境下仍具備風險。
\end{enumerate}

\subsection{8.2 未來改進方向與技術藍圖}
為了將 SentinelBox 打造為工業級的 AI 安全設施，我們規劃了以下技術演進路徑（如圖 \ref{fig:roadmap} 所示）：

\begin{itemize}
    \item \textbf{導入 eBPF CO-RE (Compile Once – Run Everywhere)}：
    計畫使用 \texttt{aya-rs} 或 \texttt{libbpf} 實作更深度的監控。eBPF 能在核心態精準抓取 Syscall 參數中的字串內容，且其非阻塞特性將能徹底解決 Seccomp User Notification 可能帶來的極微小性能干擾。

    \item \textbf{預測性沙盒溫啟動池 (Predictive Pooling)}：
    AI Agent 的執行往往具有突發性。未來計畫實作「背景暖池」，預先啟動一組已配置好 Namespace 但尚未執行代碼的沙盒。當 MCP 請求到達時，直接將進程 FD 移交，達成真正體感上的「零延遲啟動」。

    \item \textbf{結合微型虛擬機 (Firecracker/Kata)}：
    對於安全性要求極高的代碼執行任務，本系統可擴充支持微型 VM。相較於 Namespace 的共享核心隔離，VM 能提供更徹底的核心級硬體隔離，有效防禦針對 Kernel 漏洞的攻擊。

    \item \textbf{AI 意圖感知策略引擎 (Intent-Aware Policy)}：
    目前策略是靜態配置的。未來希望能由 MCP Server 根據 Agent 當前的 Task Description，動態生成最窄化權限的 Profile。例如：若任務僅是「清理 CSV 數據」，則策略引擎應自動禁止所有的網絡相關系統調用。
\end{itemize}

\section{第九章：結論}
本專案 \textbf{SentinelBox} 成功實踐了一套具備「理解力」的 Linux 安全沙盒系統。我們從 UNIX 系統程式設計的最底層出發，結合 C 語言的高效率執行、Rust 監控器的強健性、以及 TypeScript 在 MCP 協定上的靈活性，成功補齊了 AI Agent 與作業系統核心之間的「語意斷層」。

實證數據表明，語義化的錯誤反饋能顯著提升 LLM 自我修正的成功率（從 35\% 提升至 92\%），並大幅減少無意義的重試資源浪費。這不僅是一次底層技術的堆疊，更是對「AI 原生基礎設施 (AI-Native Infrastructure)」設計範式的一次大膽嘗試。

透過這個專案，我們深刻體會到：在 AI 時代，安全不應僅僅是生硬的阻斷與防火牆，而應該成為一種能夠與智能實體進行交互、導引並最終達成目標的知識資產。SentinelBox 的開發過程，讓我們在掌握 Namespaces、Cgroups 與 Seccomp 等核心技術的同時，也對未來 AI 輔助開發的安全性有了更深層次的思考。

\section{第十章：深度分析：系統呼叫攔截技術之演進}

在 Linux 系統編程中，系統調用（Syscall）攔截是所有安全防護產品的核心。為了選出最適合 AI Agent 沙盒的方案，我們對現有的四種主流技術進行了深度剖析。

\subsection{10.1 Ptrace：早期的全能攔截器}
\texttt{ptrace(2)} 是 UNIX 系統中最古老的攔截方式。它允許父行程（Tracer）完全控制子行程（Tracee）的執行，甚至可以在 Syscall 進入前修改其寄存器。
\begin{itemize}
    \item \textbf{優勢}：功能最強大，能實現「參數重寫」與「強同步攔截」。
    \item \textbf{缺陷}：每次攔截會產生兩次額外的上下文切換（Context Switch），對於高頻執行 I/O 的 AI 代碼 Alexandria 而言，性能開銷高達 200\% 以上。此外，它難以擴展到大規模並發環境。
\end{itemize}

\subsection{10.2 LSM (Linux Security Modules)：靜態強制存取控制}
如 AppArmor 或 SELinux。它們直接在核心路徑中掛載 Hook。
\begin{itemize}
    \item \textbf{局限性}：LSM 是為「長期運行的服務」設計的。AI Agent 的代碼往往是短暫的且行為不可預測，LSM 的靜態配置模式（Static Profiles）難以應對這種動態的任務變更需求，且缺乏向上層傳遞「語義細節」的通道。
\end{itemize}

\subsection{10.3 Seccomp-BPF：極速過濾器}
Seccomp 第二代利用 BPF 虛擬機在內核態直接執行過濾邏輯。
\begin{itemize}
    \item \textbf{局限性}：它的判定結果是終端性的（KILL 或 ERRNO）。雖然性能極高（幾乎零開銷），但無法實現 User Space 的動態介入。在 AI Agent 需要「知道失敗原因」的場景下，它的反饋過於生硬。
\end{itemize}

\subsection{10.4 SentinelBox 的選擇：Seccomp User Notification}
SentinelBox 採用了 Linux 5.0 引入的突破性技術。它在保留 BPF 內核過濾效能的同時，將「攔截權限」移交給 User Space 的 Rust Monitor。
這達成了一種「混合架構」：
\begin{enumerate}
    \item \textbf{Fast Path}：正常的系統調用直接放行，無額外成本。
    \item \textbf{Slow Path (Security Check)}：涉及危險的調用被暫停，並由 Monitor 進行深度分析與轉譯。
\end{enumerate}

\section{第十一章：專案部署與生產環境考量}

將 SentinelBox 從實驗室原型推向生產環境，需要解決多個維度的工程問題：

\subsection{11.1 Rootless 運作與核心攻擊面縮減}
在生產環境中，我們嚴禁使用 \texttt{--privileged} 權限執行沙盒引擎。透過 \texttt{CLONE\_NEWUSER} 與 \texttt{UID Mapping}，我們將 SentinelBox 的所有操作限制在普通用戶層級。
\begin{itemize}
    \item \textbf{優點}：即便沙盒被「穿透」，攻擊者獲得的也僅是普通用戶權限，無法觸及宿主機硬件或內核敏感區域。
    \item \textbf{配置建議}：建議在宿主機開啟 \texttt{kernel.unprivileged\_userns\_clone=1} 並限制 Namespace 的建立總數。
\end{itemize}

\subsection{11.2 Systemd Cgroup Delegation (委派機制)}
為了在 Rootless 下依然能限制內存，我們實現了對 Systemd 委派的支持。生產環境部署時，需在 \texttt{user@.service} 中設置 \texttt{Delegate=yes}，允許普通用戶管理其子資源控制組。這解決了 \texttt{/sys/fs/cgroup} 的權限衝突問題。

\subsection{11.3 性能調優：CPU Affinity 與磁碟 I/O 隔離}
\begin{itemize}
    \item \textbf{CPU Pinning}：為 Rust Monitor 分配獨立的核心，防止 Seccomp 事件處理被 AI 的計算任務干擾。
    \item \textbf{IOPS 限制}：利用 Cgroup v2 的 \texttt{io.max}，防止惡意 AI 代碼發起大規模磁碟寫入攻擊（Disk Quota Exhaustion）。
\end{itemize}

\section{第十二章：前端監控與遙測視覺化}

Dashboard 作為系統與人類交互的視圖層，其實作涉及了高效的數據流管理：

\subsection{12.1 WebSocket 與 Socket.io 的雙向推播}

\subsection{12.3 生態整合：匯出與 Kill Switch}
\begin{itemize}
    \item \textbf{Evidence Export}：支持將違規日誌匯出為 JSON/CSV，便於安全團隊事後溯源（Post-mortem Analysis）。
    \item \textbf{Kill Switch}：前端提供一鍵終止按鈕，透過 Socket 指令發回 MCP Server，最終由宿主機向該 Cgroup 發送 \texttt{SIGKILL}。
\end{itemize}

\subsection{13.1 核心原始碼導讀 (Code Walkthrough)}

\subsection{13.2 專案總結：邁向 AI 原生基礎設施}
